## 📏 ドキュメントサイズガイドライン

### 推奨ドキュメントサイズ

| ドキュメント種別 | 推奨サイズ | ページ数 | 文字数 |
|------------------|------------|----------|--------|
| 製品概要 | 簡潔 | 2-3 | 500-1,000 |
| モジュール仕様 | 詳細 | 5-10 | 1,500-3,000 |
| 機能詳細 | 中程度 | 3-5 | 800-1,500 |
| API仕様 | 技術的 | 1-2/エンドポイント | 200-400/エンドポイント |

### ドキュメント分割基準

```markdown
## 新しいドキュメントに分割すべき時:

1. **サイズ基準**
   - 単一ドキュメントが10ページを超過
   - 3,000文字を超過
   - 5つ以上の独立したセクション

2. **範囲基準**
   - 独立して開発可能な機能
   - 異なるチームが管理する領域
   - 別々のスプリントで実装予定

3. **変更頻度**
   - 頻繁に更新される部分
   - 安定した部分と分離
````

### 段階的詳細化の例

```markdown
## スプリント 1: 基本仕様

### ユーザーログイン

- 基本的なメール/パスワード認証
- セッション管理
- エラーメッセージ

## スプリント 3: 拡張詳細

### ユーザーログイン (拡張版)

認証:

- メール形式検証: RFC 5322
- パスワード要件:
  - 最小 8 文字
  - 大文字 1 文字以上、数字 1 文字以上
- パスワード表示/非表示トグル

セキュリティ:

- 最大 5 回のログイン試行
- 15 分間のアカウントロック
- 3 回失敗後に CAPTCHA

セッション:

- JWT トークン (15 分アクセス、7 日リフレッシュ)
- セキュアな httpOnly クッキー
- デバイスフィンガープリンティング
```

## 🎯 効果的な仕様書作成のコツ

### 1. 明確性

```markdown
❌ 悪い例:
"ユーザーはログインできるべき"

✅ 良い例:
"ユーザーは登録されたメールアドレスとパスワードを使用してログインできるべき。

- メール検証は大文字小文字を区別しない
- 5 回失敗後、15 分間アカウントロック
- 成功時はダッシュボードにリダイレクト"
```

### 2. 測定可能性

```markdown
❌ 悪い例:
"高速なレスポンス時間"

✅ 良い例:
"API レスポンス時間:

- 95 パーセンタイル: < 200ms
- 99 パーセンタイル: < 500ms
- 最大値: < 1000ms"
```

### 3. 完全性

```markdown
## 完全な仕様のチェックリスト

- [ ] 明確な価値を持つユーザーストーリー
- [ ] 受け入れ基準（テスト可能）

```
